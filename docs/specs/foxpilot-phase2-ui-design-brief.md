# FoxPilot 第二阶段 UI 设计简报

## 1. 文件用途

这份文件用于把 `FoxPilot` 第二阶段的界面方向一次性讲清楚，方便：

- 交给设计师做视觉和交互设计
- 交给 AI 设计工具生成页面草图或高保真稿
- 作为后续前端实现的统一参照

配套文档：

- `foxpilot-phase2-tool-architecture.md`
- `foxpilot-phase2-desktop-module-diagram.md`
- `foxpilot-phase2-cli-json-contract.md`
- `foxpilot-phase2-init-wizard-flow.md`

这份简报不讨论具体前端技术栈，只定义：

- 产品定位
- 设计目标
- 页面结构
- 信息架构
- 视觉方向
- 标准提示词
- 页面级数据契约
- Init Wizard 状态机
- 中控注册表更新与事件模型

补充约束：

- 第二阶段桌面壳固定为 `Tauri`
- 前端固定为 `React + Vite + shadcn/ui`
- 桌面端与 CLI 是两个独立入口
- 两者共用 `Runtime Core`
- 桌面端后续需要承接 `Skills` 与 `MCP` 管理
- `CLI --json` 保留为脚本化与调试接口
- 第二阶段桌面端将演进成中控平台，而不只是任务面板

## 2. 项目简介

`FoxPilot` 是一个本地优先的 AI 工程协作中控。

它的职责不是单纯记录任务，也不是通用项目管理软件，而是把以下几个对象统一到一个本地工作流里：

- 项目
- 仓库
- 任务
- 任务状态
- 运行历史
- 事件来源
- 本地环境健康状态
- `Beads` 协作
- 多平台执行协作

`FoxPilot` 当前已经完成第一阶段本地 CLI 主线，用户已经可以通过命令行完成：

- 初始化项目
- 管理多项目 / 多仓库任务
- 查看任务详情和历史
- 执行任务状态流转
- 与 `Beads` 做本地导入、导出、同步、回写、差异预览和诊断
- 安装、更新、卸载 CLI

第二阶段不再继续堆命令，而是要把“本地事件驱动协作编排”做成可视化的桌面控制台，并逐步演进成统一管理多个 agent 平台、skills 和 MCP 的中控平台。

## 3. 第一阶段 CLI 已完成范围

第二阶段 UI 设计必须建立在一个前提上：

> 第一阶段 CLI 不是“准备中”，而是已经完成并可安装使用。

也就是说，第二阶段不是重新发明任务系统，而是把已经做完的 CLI 能力做成可视化控制台。

#### 手动任务管理

### 3.2 第二阶段 UI 的承接关系

这意味着第二阶段 UI 不是重新定义这些能力，而是要把这些能力可视化：

- 把项目和仓库做成结构化视图
- 把任务和运行做成可筛选、可追踪的界面
- 把 `Beads` 协作做成可见的同步和事件面板
- 把安装、环境、健康状态做成可诊断界面
- 把基础组合安装状态做成可见的环境准备视图

### 3.3 承接方式

第二阶段 UI 不是绕过 CLI 直接重写业务逻辑。

正确做法是：

- `Tauri` 负责桌面容器与本地命令桥
- `React + Vite + shadcn/ui` 负责交互与渲染
- `FoxPilot Runtime Core` 继续负责业务动作
- `CLI --json` 作为脚本化与调试接口，不是桌面端正式主通道

### 3.4 设计时必须注意的原则

设计师在做第二阶段 UI 时，必须默认这些 CLI 能力已经存在：

- CLI 是底座
- UI 是控制台
- UI 不替代 CLI 的全部能力
- UI 优先承接最常用、最需要全局视角的能力

## 4. 背景

第一阶段的 CLI 已经把底层能力做完整，但仍然有两个明显问题：

### 4.1 可见性不足

当前很多能力只能通过命令行逐个查看，例如：

- 当前有哪些项目和仓库
- 哪些任务来自 `Beads`
- 哪些任务是本地手动创建
- 哪些任务最近执行过
- 哪些仓库同步异常
- 哪些运行失败或阻塞

这些信息分散在多个命令和输出里，适合熟悉命令行的用户，但不适合快速全局扫描。

### 4.2 协作编排缺少统一视图

第二阶段计划把 `Beads + FoxPilot + 多执行平台` 的关系做成“事件驱动协作”：

- 外部事件进入
- FoxPilot 决定下一步动作
- 不同平台执行不同阶段任务
- FoxPilot 回收结果并推进状态

如果没有统一界面，用户很难看清：

- 某个事件触发了什么动作
- 某个任务为什么进入当前状态
- 某个运行在哪一步失败
- 当前系统到底健康不健康

因此第二阶段 UI 的本质目标不是“好看”，而是：

> 把本地 AI 工程协作系统做成一个可理解、可追踪、可审计的控制台。

## 5. 第二阶段产品定位

第二阶段 UI 的定位是：

> 一个桌面优先的本地 AI 工程协作控制台。

进一步说，它会演进成：

> 一个本地 AI 协作中控平台。

它不是：

- 通用任务管理 SaaS
- 聊天机器人壳子
- 营销官网
- 纯看板应用

它更像：

- 本地工程中控台
- 任务与运行的审计台
- 事件驱动协作编排面板

## 6. 第二阶段设计目标

UI 需要帮助用户快速回答下面这些问题：

### 5.1 总览层

- 当前一共有多少项目、仓库、任务、运行
- 现在最值得处理的事情是什么
- 当前有多少阻塞、失败、待确认项

### 5.2 任务层

- 任务来自哪里
- 任务属于哪个项目 / 仓库
- 当前状态和优先级是什么
- 最近一次运行结果如何

### 5.3 运行层

- 某次执行是谁触发的
- 输入上下文是什么
- 输出结果是什么
- 改了哪些文件
- 测试结果如何
- 为什么失败或阻塞

### 5.4 协作层

- 哪些动作来自 `Beads`
- 哪些动作是 `FoxPilot` 自动编排出来的
- 哪些动作交给了哪个平台
- 哪些动作仍需要人工确认

### 5.5 环境层

- 当前本地环境是否正常
- `Beads` 是否可用
- hooks 是否配置正确
- 哪个仓库需要修复或重新初始化

## 7. 第二阶段不做什么

为了避免 UI 一开始就失控，以下方向不作为这一版核心目标：

- 不先做远程服务管理后台
- 不先做多人实时协作界面
- 不先做移动端优先界面
- 不先做大而全的工作流设计器
- 不先做聊天窗口作为主界面

## 8. 推荐平台与布局

### 7.1 平台

这一版 UI 建议：

- 桌面优先
- 大屏笔记本和显示器优先
- 允许后续自适应到较窄窗口
- 技术承载以 `Tauri` 桌面窗口为前提

### 7.2 布局

推荐使用：

- 左侧导航
- 中央主工作区
- 右侧上下文详情面板
- `Init Wizard` 作为条件流程入口，不进入常驻一级导航

这套布局适合高信息密度产品，因为它能同时承载：

- 导航
- 列表 / 表格
- 详情 / 侧边上下文

而不会频繁跳页丢失上下文。

## 9. 信息架构

推荐先设计下面 7 个常驻一级页面。

### 8.1 Dashboard 总览

用于回答“现在系统整体怎么样”。

建议包含：

- 项目总数
- 仓库总数
- 进行中任务
- 阻塞任务
- 最近运行
- 最近同步
- 环境告警

### 8.2 Workspace 项目与仓库

用于回答“项目和仓库现在是什么状态”。

建议包含：

- 项目列表
- 每个项目下的仓库清单
- 每个仓库的最近同步状态
- `doctor` 状态
- `hooks` / `init-beads` 状态
- 该仓库关联任务数量

### 8.3 Tasks 任务中心

这是最核心页面之一。

建议以表格为主，因为表格最适合表达：

- 任务标题
- 来源
- 状态
- 优先级
- 阶段
- 角色
- 平台
- 目标仓库
- 更新时间
- 是否有运行历史

这页允许后续再补看板视图，但第一版仍建议表格优先。

### 8.4 Run Detail 运行详情

这是第二个最核心页面。

用于展示一次运行的完整上下文：

- 运行来源
- 触发事件
- 输入上下文
- 执行摘要
- 输出结果
- 变更文件
- 测试结果
- 错误信息
- 后续建议

### 8.5 Event / Hooks Timeline 事件时间线

用于回答“事件是怎么把系统推进到当前状态的”。

建议包含：

- 事件来源
- 事件时间
- 事件类型
- 触发动作
- 动作结果
- 是否成功
- 是否需要人工确认

### 8.6 Control Plane 中控入口

用于回答：

- 当前有哪些 agent 平台
- 哪些 skills 已安装、已损坏、可修复
- 哪些 MCP 已配置、已降级、需重启

建议拆成三个子页：

- Platforms
- Skills
- MCP

### 8.7 Settings / Health 设置与环境健康

用于回答“系统配置和环境是否正常”。

建议包含：

- 当前 CLI 版本
- 安装方式
- 当前语言
- 工作区路径
- `Beads` 环境状态
- hooks 状态
- `doctor` 结果
- 同步能力状态

## 10. 关键对象与数据标签

为了让设计稿和后续实现保持一致，界面里建议统一使用以下核心对象：

- 项目 `project`
- 仓库 `repository`
- 任务 `task`
- 任务来源 `source`
- 任务状态 `status`
- 优先级 `priority`
- 阶段 `stage`
- 角色 `role`
- 平台 `platform`
- 运行 `run`
- 事件 `event`
- 环境健康状态 `health`

常见来源建议明确区分：

- `local`
- `beads`

常见状态建议显式区分：

- `todo`
- `analyzing`
- `implementing`
- `verifying`
- `done`
- `blocked`
- `cancelled`

## 11. 视觉方向建议

### 10.1 关键词

推荐关键词：

- 工程控制台
- 高信息密度
- 稳定
- 克制
- 专业
- 可扫描
- 可审计

### 10.2 不推荐的关键词

不建议使用：

- 紫色 AI 默认风格
- 过度玻璃拟态
- 空白过多的营销 SaaS 风
- 全页面只靠卡片堆叠
- 过强娱乐化色彩

### 10.3 颜色建议

建议方向：

- 主背景：石墨灰 / 浅暖灰
- 主文本：高对比冷白 / 深墨灰
- 强调色：青蓝或冷绿
- 状态色：
  - 成功：绿色
  - 告警：黄色 / 琥珀色
  - 失败：红色
  - 次要：中性灰

### 10.4 字体建议

建议组合：

- 主字体：`IBM Plex Sans`
- 等宽字体：`JetBrains Mono`

原因：

- 工程感强
- 信息密度高时可读性好
- 代码、路径、命令显示更自然

### 10.5 图标方向

建议准备这些类型的图标：

- 终端
- 仓库
- 分支
- 任务
- 运行
- 事件
- 同步
- 告警
- 设置
- 健康检查

## 12. 关键交互建议

### 11.1 任务中心

建议：

- 表格为主
- 支持按来源、状态、仓库、优先级筛选
- 支持快速查看右侧详情
- 支持看到最近一次运行结果

### 11.2 运行详情

建议：

- 头部显示运行状态和来源
- 中间显示输入 / 输出 / 变更 / 测试
- 右侧显示相关任务和相关仓库
- 错误和阻塞原因必须显眼

### 11.3 时间线

建议重点展示：

> 事件 -> 编排动作 -> 执行结果

让用户看得懂“为什么发生了这一步”，而不是只看最后结果。

## 13. 设计交付重点

如果设计资源有限，建议优先把下面 3 页做精：

- Dashboard 总览
- Tasks 任务中心
- Run Detail 运行详情

原因是这三页最能定义产品气质，也最能支撑后续前端实现。

## 17. Project Init Wizard 细化设计

这一节用于把 `Project Init Wizard` 细化成可以直接出图、出交互稿的结构。

### 16.1 页面定位

`Project Init Wizard` 是项目级初始化页面。

它不负责系统级基础依赖安装，因为那部分已经在安装阶段完成。它的职责是：

- 识别当前项目
- 让用户确认项目接入方式
- 选择项目协作 profile
- 展示阶段 / 角色 / 平台解析结果
- 汇总即将写入的项目级配置

### 16.2 页面目标

用户进入这个页面后，应该能非常清楚地知道：

- 系统识别到了什么项目
- 当前项目会被怎么接入
- 会启用哪套协作 profile
- 系统为哪些阶段解析了什么角色和平台
- 最终会写入哪些配置

### 16.3 建议布局

建议采用三栏结构：

- 左栏：步骤导航
- 中栏：当前步骤主内容
- 右栏：上下文说明 / 当前选择摘要

#### 左栏步骤建议

按顺序展示：

1. 项目扫描
2. Profile 选择
3. 平台解析
4. 初始化确认
5. 初始化完成

#### 中栏作用

中栏是当前步骤主舞台，用于承载：

- 表单
- 识别结果
- 选择器
- 说明文案
- 总结信息

#### 右栏作用

右栏建议始终存在，用于展示：

- 当前项目路径
- 项目类型
- 仓库数量
- 当前已选 profile
- 当前平台解析结果
- 当前启用能力摘要

这样用户在切换步骤时不会丢失上下文。

## 18. 字段清单

### 17.1 Step 1：项目扫描

建议展示的字段：

- 当前工作区路径
- 项目名称
- 项目类型
- 仓库数量
- 主仓库名称
- 识别信号列表
- 当前语言栈
- 扫描状态

#### 字段说明

**当前工作区路径**

用于明确当前接管的目录位置，例如：

```text
/Users/zhouping/code/my-project
```

**项目名称**

默认由目录名或现有配置推断，允许用户在后续步骤中修改。

**项目类型**

例如：

- Node / TypeScript
- Python
- Go
- Rust
- Mixed Workspace
- Docs Workspace

**仓库数量**

如果是单仓库项目，显示 `1`。  
如果是多仓库工作区，显示总数，并可点开预览。

**识别信号列表**

用于解释“为什么系统判定这是这个项目类型”，例如：

- 检测到 `package.json`
- 检测到 `pnpm-lock.yaml`
- 检测到 `apps/` 与 `packages/`

**当前语言栈**

用于后续平台解析的依据说明，例如：

- TypeScript
- JavaScript
- Python

### 17.2 Step 2：Profile 选择

建议展示的字段：

- Profile 名称
- Profile 类型标识
- 推荐标记
- 启用能力
- 不启用能力
- 适用场景
- 当前选择状态

#### 推荐的三个 Profile

**default（推荐）**

启用：

- FoxPilot
- Beads
- superpowers

适用场景：

- 正常开发项目
- 需要完整本地协作能力

**collaboration**

启用：

- FoxPilot
- Beads

适用场景：

- 先接任务来源和本地同步
- 暂不启用 superpowers

**minimal**

启用：

- 仅 FoxPilot

适用场景：

- 先只使用本地任务中控
- 受限环境或轻量接入

### 17.3 Step 3：平台解析

建议展示的字段：

- 解析结果状态
- 阶段清单
- 角色映射
- 当前推荐平台
- 已检测到的平台列表
- 未检测到的平台列表
- 判定依据
- 是否允许手动覆盖

#### 核心规则

这一层不让用户“先选 Codex”，而是先展示：

- 系统检测到了什么平台
- 系统为什么这么判断
- 当前各阶段的推荐平台是什么

#### 探测状态建议

- 已检测到，可启用
- 未检测到，不可用
- 已检测到，但当前阶段不推荐
- 已检测到多个，已自动推荐一个

#### 判定依据建议

展示为可读说明，而不是纯日志，例如：

```text
已检测到 Codex 与 Claude Code
判定依据：
- 本机存在 codex 命令
- 本机存在 claude code 命令
- 当前项目为 Node / TypeScript 项目
- design 阶段优先分配给 codex
- implement 阶段优先分配给 claude_code
```

### 17.4 Step 4：初始化确认

建议展示的字段：

- 项目路径
- 项目名称
- 项目类型
- 仓库清单摘要
- 选中 profile
- 阶段 / 角色 / 平台 摘要
- 将启用的能力
- 将写入的配置文件

#### 将写入的配置文件建议显式展示

- `~/.foxpilot/foxpilot.config.json`
- `<project-root>/.foxpilot/project.json`

如果有额外环境配置，也建议在这里展示。

### 17.5 Step 5：完成页

建议展示的字段：

- 初始化完成状态
- 已接管项目名称
- profile 结果
- 平台解析结果
- 下一步建议动作
- 快捷操作入口

## 20. Control Plane 页面设计重点

这一节用于保证第二阶段 UI 不会只停留在任务控制台，而是能真正承接中控平台。

### 20.1 Platforms 页面

这个页面的目标不是展示“平台名字列表”，而是回答：

- 当前有哪些 agent 平台可用
- 每个平台当前是否健康
- 每个平台支持哪些能力
- 当前项目或当前阶段会倾向使用哪个平台

建议表格字段：

- 平台名称
- 状态
- 版本
- 支持能力
- 推荐阶段
- 最近检查时间

右侧面板建议展示：

- 平台详情
- 探测理由
- 失败原因
- 可执行动作

### 20.2 Skills 页面

这个页面的目标是：

- 看清技能安装状态
- 看清哪些技能损坏或可修复
- 做启用 / 禁用 / repair

建议表格字段：

- 技能名称
- 来源
- 版本
- 状态
- 安装路径
- 最近检查时间

右侧面板建议展示：

- 技能详情
- manifest / 说明文件位置
- doctor 结果
- repair 建议

### 20.3 MCP 页面

这个页面的目标是：

- 看清当前有哪些 MCP
- 哪些 MCP 正常、降级、不可用
- 哪些 MCP 需要重启或修复

建议表格字段：

- Server 名称
- 状态
- 配置来源
- 命令
- 最近检查时间

右侧面板建议展示：

- 配置路径
- 命令与参数
- 最近错误
- doctor / repair / restart 入口

### 20.4 中控平台页的共同风格

这三页建议统一：

- 表格为主
- 右侧详情为辅
- 顶部保留健康摘要条
- 顶部保留批量操作 / refresh / doctor 入口

不要做成：

- 纯卡片堆叠
- 只会显示状态徽章、不显示可执行动作
- 和 Settings 混成一个大杂烩页面

建议完成页至少提供一个主动作：

- 查看任务中心

以及两个次动作：

- 查看项目详情
- 运行环境检查
