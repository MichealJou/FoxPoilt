# FoxPilot 第二阶段 Events 页面密度策略

## 1. 文档目的

这份文档只定义一件事：

> 第二阶段 `Events` 页面应该承载多高的信息密度、哪些信息放在组标题、哪些信息留到展开后，避免一打开就是难以扫描的事件墙。

如果没有这层策略，后面很容易出现：

- 每组时间线一展开就塞满几十个字段
- 页面要么过于稀疏，要么过于拥挤
- 组标题和展开内容重复

## 2. 定位

这份策略不是：

- 时间线分组规则
- 视觉样式稿
- 事件类型定义

它是：

> `Events` 页信息密度和层次分配规则。

## 3. 第一原则

第二阶段 `Events` 页必须固定：

```text
组标题负责扫描
展开内容负责解释
原始字段按需下钻
```

## 4. 第一批组标题信息

建议每个 `TimelineGroup` 标题层只展示：

```text
title
status
timelineKind
rootTarget
最近时间
事件数量
一句 summary
```

不要在标题层直接展示：

- 完整错误文本
- 所有事件名
- 全部 artifact
- 全部 participant

## 5. 第一批展开层信息

展开后再展示：

```text
事件序列
关键动作结果
handoff 摘要
session 摘要
推荐跳转
```

## 6. 原始字段下钻规则

第二阶段不建议在 `Events` 第一批直接露出全部原始 payload。

建议只在第三层下钻里按需展示：

```text
event payload 摘要
error detail 原文
artifact 列表
refreshHints
```

## 7. 扫描优先级

页面一眼必须先看清：

```text
失败链
阻塞链
待确认链
最近完成链
```

所以：

- 失败 / 阻塞态必须有明显视觉优先级
- 低优成功链不要压过异常链

## 8. 与 Runs / Tasks 的关系

`Events` 页不是为了替代 `Runs` 和 `Tasks`。

它的职责是：

```text
回答“为什么会到这里”
```

而不是：

```text
把 Run Detail 和 Task Detail 再复制一遍
```

## 9. 第一批范围控制

第二阶段第一批先不做：

- 无限展开日志
- 大段原始 JSON
- 多栏复杂泳道

先固定：

```text
稳定标题层
稳定展开层
稳定第三层原始下钻
```

## 10. 审核点

你审核这份策略时，重点看：

```text
1  是否接受 Events 页标题层只承载扫描信息，不承载全部字段
2  是否接受 原始 payload 只在第三层下钻按需展示
3  是否接受 失败 / 阻塞 / 待确认 链路拥有更高视觉优先级
4  是否接受 Events 页只解释链路，不复制 Task / Run 详情页
```
