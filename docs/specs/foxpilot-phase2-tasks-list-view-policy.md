# FoxPilot 第二阶段 Tasks 列表视图策略

## 1. 文档目的

这份文档只定义一件事：

> 第二阶段 `Tasks` 页面应该如何组织列表视图、筛选、排序和摘要列，避免它变成一张字段越堆越多的超级表。

如果没有这层策略，后面很容易出现：

- 表格列越来越多
- 任务列表和详情页分工不清
- `stage / role / platform / source` 等关键字段表现不一致

## 2. 定位

`Tasks` 页面不是：

- 完整详情页
- Dashboard 的放大版
- 纯数据库表查看器

它是：

> 第二阶段任务扫描、筛选、排序和进入详情的主工作台。

## 3. 第一原则

第二阶段 `Tasks` 页面必须固定：

```text
列表优先
筛选先于操作
详情深挖而不是列表堆字段
```

## 4. 第一批视图模式

建议第二阶段先只正式支持：

```text
Table View
```

可以预留后续：

```text
Board View
```

但第一批不要同时实现两套主视图。

## 5. 第一批核心列

建议第二阶段表格固定为这些核心列：

```text
标题
来源
状态
优先级
阶段
角色
平台
仓库
最近更新时间
最近运行状态
```

## 6. 为什么列数要克制

任务列表的职责是：

```text
快速扫描
快速筛选
快速定位
```

不是：

```text
一次性看完全部上下文
```

所以这些信息不应直接塞进表格主列：

- handoff 产物详情
- execution session 全字段
- 完整错误文本
- 历史事件流

这些都应该留给右侧面板和详情页。

## 7. 第一批筛选维度

建议第二阶段固定：

```text
source
status
priority
stage
role
platform
repositoryId
hasRecentFailure
needsConfirmation
```

## 8. 第一批排序维度

建议第二阶段固定：

```text
updatedAt
priority
status
stage
source
```

默认排序建议为：

```text
updatedAt desc
```

## 9. 右侧面板与详情页分工

`Tasks` 页应明确：

- 表格负责扫描
- 右侧面板负责解释当前选中任务
- Task Detail 负责完整推进

不要让：

- 表格承担详情职责
- 右侧面板承担完整详情职责

## 10. 第一批行级动作

建议第二阶段行级动作保持克制，只放：

```text
查看详情
打开最近运行
跳转平台
轻量展开右侧面板
```

不建议直接在列表中塞入：

```text
task.reassign
复杂批量推进
复杂 destructive 操作
```

## 11. 批量动作边界

第二阶段 `Tasks` 页第一批建议只做：

```text
批量选择预留
批量动作后置
```

也就是说，列表先支持多选模型，但不急着开放复杂批量写操作。

## 12. 与 Dashboard 的关系

Dashboard 提供：

```text
任务摘要
高优入口
```

Tasks 提供：

```text
完整任务扫描与筛选面
```

所以 Dashboard 不应替代 Tasks。

## 13. 第一批范围控制

第二阶段第一批先不做：

- 自定义列
- 多视图同步
- 表格内复杂编辑
- 无限列扩展

先固定：

```text
稳定核心列
稳定筛选
稳定排序
稳定右侧面板联动
```

## 14. 审核点

你审核这份策略时，重点看：

```text
1  是否接受 Tasks 第一批以 Table View 为唯一正式主视图
2  是否接受标题 / 来源 / 状态 / 优先级 / 阶段 / 角色 / 平台 / 仓库 / 更新时间 / 最近运行状态 作为核心列
3  是否接受 handoff / session / 错误详情 不直接进表格主列
4  是否接受 Tasks 页第一批以扫描与筛选为主，不急着开放复杂批量写动作
```
