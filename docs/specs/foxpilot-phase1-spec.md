# FoxPilot 第一阶段规格说明

## 1. 产品定位

FoxPilot 是一个面向本地开发环境的多项目任务中控工具。

第一阶段先完成产品模型和规则设计，不直接进入完整界面与复杂执行编排。

FoxPilot 的长期目标包括：

- 管理多个本地开发项目
- 统一接管任务
- 支持 `Beads + Superpowers + Codex` 协作
- 后续扩展为桌面端可视化中控台

## 2. 第一阶段目标

第一阶段重点是定义清楚：

- 工作区根目录如何管理
- 项目如何定义
- 仓库如何挂载到项目下
- 任务如何归属与流转
- `Beads + Superpowers + Codex` 在系统中的职责
- CLI 安装阶段如何接管基础依赖组合

本阶段不做：

- 完整桌面界面
- 多代理工种系统
- 向量记忆
- Aider 深度接入

## 3. 核心对象模型

### 3.1 工作区根目录

工作区根目录是本地扫描起点，例如：

- `/Users/program/code`

系统允许配置一个或多个工作区根目录。FoxPilot 启动时只扫描这些目录，不做全盘扫描。

### 3.2 项目

项目默认表示一个根工作区，而不是单个具体仓库。

原因：

- 更符合跨仓库任务场景
- 更适合任务中控产品定位
- 后续支持项目级任务、仓库级任务和跨仓库任务

项目支持两种接入方式：

- 自动扫描发现
- 手动添加

### 3.3 仓库

仓库是项目下的具体代码仓库或子仓库。

仓库用于：

- 标记任务目标范围
- 提供扫描上下文
- 定位后续执行边界

仓库不是任务默认归属层，任务默认归属在项目层。

### 3.4 任务

任务默认挂在项目层。

任务可以额外指定：

- 目标仓库
- 目标模块
- 目标目录

这样可以同时支持：

- 项目级任务
- 仓库级任务
- 跨仓库任务

## 4. 任务来源

第一阶段任务来源固定为三类：

- 手动创建
- 从 Beads 同步
- 项目扫描后生成建议任务

其中：

- 手动创建用于用户主动登记
- Beads 同步用于承接执行层任务
- 项目扫描建议用于初始化项目上下文

## 5. 任务状态流转

第一阶段采用 AI 工作流型状态：

- `待处理`
- `分析中`
- `待确认方案`
- `执行中`
- `待确认结果`
- `已完成`
- `已阻塞`
- `已取消`

推荐流转主路径：

1. 待处理
2. 分析中
3. 待确认方案
4. 执行中
5. 待确认结果
6. 已完成

异常路径：

- 任意阶段可以转 `已阻塞`
- 任意未完成阶段可以转 `已取消`

## 6. 第一阶段协作模型

### 6.1 Beads

职责：

- 任务底座
- 依赖管理
- ready / blocked / doing / done 状态承载

### 6.2 Superpowers

职责：

- 方法层
- 复杂任务推进方式
- spec / plan / review 的流程约束

### 6.3 Codex

职责：

- 分析任务
- 执行任务
- 回写任务结果
- 后续作为总调度入口

### 6.4 Aider

第一阶段不纳入核心链路，只保留为后续可选执行器。

### 6.5 基础组合安装

第一阶段补充一个固定基础组合：

- `Beads`
- `Superpowers`

这个组合的定位不是“项目级 profile”，而是“安装阶段基础能力”。

也就是说，在第一阶段产品边界里：

- 安装阶段负责接管基础组合
- `foxpilot init` 负责接管项目
- `Codex` 继续按执行器视角处理，而不是放进基础组合里

基础组合安装建议遵循以下原则：

- 缺失时优先按官方方式安装
- 安装后要有版本或健康检查
- 要记录安装结果和安装来源
- 不把 `Codex` 桌面端纳入自动安装目标

这样做的目的是：

- 让用户在第一次安装 FoxPilot 后就具备最小协作底座
- 避免等到 `init` 阶段才发现缺少关键依赖
- 让后续项目级初始化只关注“项目怎么接入”，而不是“机器还缺什么工具”

## 7. 第一阶段任务展示方式

第一阶段默认采用表格列表视图，而不是先做看板。

原因：

- 更适合多项目筛选
- 更适合多条件过滤
- 更适合第一阶段先把数据模型做稳

后续再扩展看板视图。

## 8. 第一阶段最小数据结构

建议第一版至少包含这些实体：

### 8.1 工作区根目录

- `id`
- `name`
- `path`
- `enabled`
- `createdAt`
- `updatedAt`

### 8.2 项目

- `id`
- `name`
- `rootPath`
- `workspaceRootId`
- `sourceType`（扫描发现 / 手动添加）
- `status`（待接管 / 已接管 / 停用）
- `createdAt`
- `updatedAt`

### 8.3 仓库

- `id`
- `projectId`
- `name`
- `path`
- `repoType`
- `enabled`
- `createdAt`
- `updatedAt`

### 8.4 任务

- `id`
- `projectId`
- `title`
- `description`
- `sourceType`
- `status`
- `priority`
- `requiresPlanConfirm`
- `createdAt`
- `updatedAt`

### 8.5 任务目标

- `id`
- `taskId`
- `repoId`
- `targetType`（仓库 / 模块 / 目录）
- `targetValue`
