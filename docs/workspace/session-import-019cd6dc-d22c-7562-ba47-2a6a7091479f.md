## 3. 已收敛的通用结论

### 3.1 状态与历史记录要分开

这次会话反复收敛到一个稳定模式：

- 主表只保留“当前有效状态”与“最近一次摘要”
- 历史过程单独落历史表
- 当前态用于列表、筛选、统计
- 历史表用于追溯、审计、多次提交记录

可抽象为：

- `主实体`：当前状态
- `草稿实体`：未正式生效内容
- `提交/运行记录实体`：每次动作的历史记录

这个模式对 FoxPilot 后续很有参考价值，尤其适合：

- `task` 只保留当前状态
- `task_run` 记录分析、执行、验证等历史
- 若后续需要，可再补 `task_draft` 或 `plan_draft`

### 3.2 状态流转必须显式定义

旧会话对状态口径收得很严，重点不是“字段多”，而是：

- 状态集合固定
- 主路径固定
- 异常路径固定
- 哪些状态允许再次编辑或再次提交，要提前写死规则

这对 FoxPilot 第一阶段也适用：

- 任务状态不能只列名字，必须同时给出主路径和异常路径
- 哪些状态允许重新分析、重新执行、重新确认，需要单独列规则

### 3.4 编排层与单模块职责要分开

旧会话后半段最稳定的一条规则是：

- 一个 Service 只服务一个业务语义或一个数据域
- 跨模块依赖只允许出现在 Biz/编排层
- 编排层只做串调用、顺序控制、失败处理、回写关系
- 编排层不直接写 mapper/sql

这对 FoxPilot 也很重要。后续如果进入实现阶段，建议保持：

- `repository` 相关逻辑只处理仓库域
- `task` 相关逻辑只处理任务域
- `executor`、`sync`、`scan` 的跨域联动放在编排层
