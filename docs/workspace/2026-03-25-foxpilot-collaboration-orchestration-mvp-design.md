# FoxPilot Collaboration Orchestration MVP Design

## 1. 背景

当前工作区已经完成：

- `foxpilot init` / `fp init`
- 手动任务创建、列表、详情、历史
- 状态更新、执行器切换、优先级调整
- 下一条任务选择
- 扫描建议任务生成

也就是说，FoxPilot 已经具备“本地任务底座”和“人工管理闭环”。

下一步最自然的主线，不再是继续堆更多单点命令，而是把第一阶段规格里真正有产品意义的协作能力落成下一版设计：

- `Beads` 如何把任务送进 FoxPilot
- FoxPilot 如何区分“同步层”和“编排层”
- `beads_sync` 来源任务如何与手工任务并存

## 2. 设计目标

这一版设计只解决 4 个问题：

1. 给 `Beads` 同步预留一个稳定的本地入口
2. 明确同步层和编排层的边界
3. 明确 `beads_sync` 任务的最小去重规则
4. 明确下一步实现应该先落哪个 CLI 闭环

## 3. 不做什么

这一版设计明确不做：

- 不直接接真实 `Beads` 网络接口
- 不引入桌面端协作面板
- 不引入自动执行策略或复杂状态机
- 不引入 `beads_mapping` 新表
- 不做跨来源任务自动合并

原因很直接：

- 真实外部同步一旦接入，就会把当前 MVP 从“本地可控”变成“外部依赖驱动”
- 当前最缺的不是外部连通，而是边界规则
- 先把本地导入语义和幂等规则定清楚，后面接真实同步才不会返工

## 4. 核心边界

### 4.1 同步层

同步层只负责三件事：

1. 接收外部任务快照
2. 把外部字段映射为 FoxPilot 可接受的本地任务输入
3. 执行幂等 upsert 或跳过

同步层不负责：

- 自动推进任务状态
- 自动切换执行器
- 自动触发 `task_run`
- 自动分配优先级策略

### 4.2 编排层

编排层只负责跨域规则：

- 外部同步任务进入后，默认由谁接手
- 是否需要创建扫描建议或分析任务
- 哪些状态允许人工接管
- 哪些来源之间允许建立引用关系

编排层不直接写 SQL，不直接解析外部协议。

### 4.3 任务域

任务域保持当前职责：

- 持久化 `task`
- 持久化 `task_target`
- 持久化 `task_run`
- 提供查询、筛选、更新当前态的能力

任务域不知道“这个任务是怎么从外部来的”，只知道它的 `source_type` 是什么。

## 5. 建议的数据映射

当前阶段先不新增表，直接复用已有 `task` 与 `task_target`：

- `task.source_type = beads_sync`
- `task.status` 初始建议映射为：
  - ready -> `todo`
  - doing -> `executing`
  - blocked -> `blocked`
  - done -> `done`
- `task.current_executor` 初始建议为：
  - `beads`
- `task.execution_mode` 初始建议为：
  - `manual`
- `task.requires_plan_confirm` 初始建议为：
  - `1`

这样做的目的不是“精确复制 Beads”，而是把外部任务先稳定落到 FoxPilot 当前模型里。

## 6. 幂等规则

第一版建议不用新增 `beads_mapping` 表，而是采用“外部来源键 + 本地任务来源约束”的最小方案：

### 6.1 最小外部标识

在导入命令里，要求每条外部任务至少提供：

- `externalSource = beads`
- `externalTaskId`

### 6.2 当前阶段落地方式

第一版先把外部标识折叠进任务描述范围之外的可控字段：

- 作为导入命令输入的唯一键参与查询
- 如果命中已有 `beads_sync` 任务，则更新当前态
- 如果未命中，则创建新任务

这里的关键结论是：

- 幂等判断属于同步层
- 当前态更新属于任务域
- “是否需要新建 task_run” 先不在同步入口里做

### 6.3 为什么先不加 `beads_mapping`

因为第一版还没有真实 `Beads` API，也没有多来源同步需求。
如果现在先建映射表，极大概率会把字段设计建立在错误假设上。

## 7. 下一步最小实现入口

下一步建议先做一个本地可验证的导入命令，而不是直接做远程同步：

### 7.1 建议命令

```bash
foxpilot task import-beads --file <json-file>
fp task import-beads --file <json-file>
```

### 7.2 输入假设

`json` 文件里每一条记录至少包含：

- `externalTaskId`
- `title`
- `status`
- `priority`
- `repository`

### 7.3 MVP 输出

命令输出只需要回答：

- 创建了多少条
- 更新了多少条
- 跳过了多少条
- 哪些记录因为字段不合法被拒绝

### 7.4 为什么推荐这条命令

因为它有几个明显优势：

- 不依赖外部网络
- 可以用固定样例文件做测试
- 能把同步层、任务域、编排层的边界先走通
- 后面接真实 `Beads` 时，只需要替换“快照来源”，不用推倒导入语义

## 8. 推荐实现顺序

### 8.1 第一批

- 先写 `task import-beads` 的任务登记文档
- 再写实现计划
- 最后用 TDD 落代码

### 8.2 第二批

- 增加样例导入文件结构
- 增加幂等测试
- 增加跨项目隔离测试

### 8.3 第三批

- 再考虑是否需要把外部同步记录从命令层抽成独立同步服务

## 9. 当前结论

当前最合理的下一主线是：

1. 不再继续补零散手动命令
2. 先把协作编排边界文档定住
3. 下一条真正该实现的命令，优先选择 `task import-beads`

这样 FoxPilot 才会从“本地任务 CLI”继续往“协作中控”推进，而不是停留在单机任务本。
